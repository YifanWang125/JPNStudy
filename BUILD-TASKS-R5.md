## B. Study Pet（v1.1）复审发现
总评：实现很用心、教学设计聪明（日记=阅读练习、进化=新语体、性格=词汇领域、成长绑定真实进步而非签到；XSS 安全——AI/用户文本都过 esc/toRuby；无 setInterval 泄漏；离线衰减有 48h 上限）。问题集中在性能与产品取舍：

### B1【代码·性能，应修】动画 rAF 永不停止
`js/pet.js`：`stopAnim()` 定义了但**从未被调用**，也没有 `visibilitychange`/可见性判断。`requestAnimationFrame(draw)` 以 ~60fps **持续重绘** pet canvas——即使在非 home 页、或浏览器标签隐藏时也在跑（隐藏的 canvas 仍每帧 drawCreature）。移动端会无谓耗电/占 CPU。
- 改：仅当 home 可见或聊天/日记 overlay 打开时跑动画；切走/`document.hidden` 时 `stopAnim()`；回来再 `startAnim()`。（pet 要 idle 摆动，所以不是删动画，而是按可见性暂停。）

### B2【产品·请作者拍板】宠物“永久死亡”是否适合一个鼓励型学习工具
pet 可生病、HP 归零后**永久死亡**（进 memorial）。衰减虽有 48h 上限+24h 生病宽限，较温和，但一个忙碌的成人学习者长时间没开，仍可能回来看到宠物濒死/死亡——对“激励学习”的初衷可能**适得其反**（愧疚、被惩罚感）。
- 建议（任选其一，倾向前者）：长时间离开改为「冬眠/おやすみ」而非死亡；或把“永久死亡”做成**默认关闭、硬核可选**；或保证可“复活/再迎”无损失感。当前虽偏温和，但风险方向值得你定。

### B3【产品·home 布局失衡，建议调】宠物把核心「继续学习」挤到首屏之外
现在 home 顶部是较大的宠物卡（+欢迎横幅），桌面端把「▶ 继续学习 · Day N」压到下方，**移动端整屏几乎全是宠物蛋+横幅，学习 CTA 完全在首屏之下**（要滚动才看到）。学习类应用的主行动应当优先可见。（设计文档 §F 已自列为 open issue。）
- 建议：home 首屏先保证「继续学习」可见；桌面把 pet 作为侧栏/次要列，移动端 pet 折叠或置于 CTA 之下。

### B4【小】首次欢迎横幅没提到宠物
`app.js` 的 introBanner 列了 真人音色/AI/速记/浅深色，却没提这次的主角——宠物。可加一条“🥚 学习会孵化&养成你的言霊宠物”。

### B5【可选·成本提示】AI 调用频率
`onStudy()` 每次标记完成/交卷都会触发 `generateStudyLine()`（haiku 一次调用）；日记/聊天另计。BYOK + haiku 很便宜，但可在文档里告知用户“宠物会用少量 API 额度”。
